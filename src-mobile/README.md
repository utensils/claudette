# claudette-mobile

The Claudette iOS / Android client. Thin WSS remote-control app — it pairs with a running Claudette desktop or headless `claudette-server` and drives chat sessions over the same wire protocol the desktop's Remote Workspaces feature uses.

## Status

Phase 4 of the iOS foundation work. Today this directory contains the Rust crate, Tauri config, and capability manifest needed to compile and boot an empty webview. Screens (pair, workspace list, chat, etc.) land in Phases 5–8.

## Setup (iOS, one-time)

Tauri 2's iOS support uses Apple's toolchain, so device builds require:

1. **macOS** with [Xcode](https://developer.apple.com/xcode/) installed (Xcode 15+ recommended).
2. An [Apple Developer Program](https://developer.apple.com/programs/) account if you want to deploy to a physical iPhone (free account works for the simulator and ad-hoc on-device testing for 7 days).
3. The iOS Rust targets:
   ```bash
   rustup target add aarch64-apple-ios aarch64-apple-ios-sim x86_64-apple-ios
   ```
4. The Xcode project scaffolding, generated by Tauri:
   ```bash
   cd src-mobile
   cargo tauri ios init
   ```
   This creates `src-mobile/gen/apple/` containing the Xcode project, Swift wrapper, and `Info.plist`. The generated files are committed to the repo — when you regenerate, prefer `cargo tauri ios init --reinstall` so existing customizations (signing identity, capabilities) survive.

## Running

```bash
# iOS simulator (camera-less; use the Advanced/paste fallback for pair tests):
cargo tauri ios dev

# Physical device (requires Apple Developer signing identity):
cargo tauri ios dev --host

# Desktop fallback for fast UI iteration (no Xcode needed):
cargo tauri dev --bin claudette-mobile
```

The desktop fallback boots the same webview in a 390×844 window so you can iterate on the mobile UI without an Xcode round-trip. It's not a substitute for on-device testing (camera, Keychain, safe-area insets all behave differently) but it's much faster for layout work.

## Workspace layout

| Path | What's there |
|---|---|
| `Cargo.toml` | Crate manifest. Depends on `claudette` with `default-features = false` so the agent runtime / Pi sidecar / heavy bits stay out of the mobile binary. |
| `src/lib.rs` | Tauri runtime entry. The `#[tauri::mobile_entry_point]` macro exposes a FFI symbol the Swift / JNI wrappers launch. |
| `src/main.rs` | Desktop-fallback entry point — `cargo tauri dev --bin claudette-mobile`. |
| `src/state.rs` | `ConnectionManager` holding active `Transport` instances per paired server. |
| `src/commands.rs` | Tauri-invokable commands. Phase 4: just `version` (a liveness probe). |
| `tauri.conf.json` | Tauri config — bundle identifier `dev.claudette.mobile`, iOS minimum version 16.0. |
| `capabilities/default.json` | Webview permission manifest. Phase 4 grants `core:default` + `barcode-scanner:default`. |
| `icons/` | App icon. Seeded from the desktop `src-tauri/icons/icon.png`; replace with mobile-specific art when ready. |
| `gen/apple/` | **Not in git initially.** Generated by `cargo tauri ios init` — commit after first run on your machine. |

## What does NOT run on the phone

Spelled out in detail in [`site/src/content/docs/features/mobile-app.mdx`](../site/src/content/docs/features/mobile-app.mdx), but the short version:

- No `claude` subprocess. The phone is a WSS client; agents run on the paired host.
- No SQLite. Workspace metadata is fetched from the host on each session.
- No Git, no PTY, no MCP servers, no voice. All of those are desktop / server features.

This makes the mobile binary fundamentally small and lets the Rust side share the `claudette` lib's `transport` and `model` modules without dragging in the rest of the agent stack.
